# Function Checker

用于对待复用的 Linux 内核模块代码执行跨 `.ko` / `vmlinux` 的静态闭包分析，识别用户态复用前最常见的危险点：

- 特权指令与控制寄存器访问
- 绝对地址重定位
- 跨节区数据依赖
- 跨模块 / 跨 `vmlinux` 调用闭包
- 间接控制流、函数指针与 Retpoline 包装跳转
- 编译器插桩、栈保护与 Sanitizer 残留

## 1. 设计目标

这个工具的目标不是“替你下最终结论”，而是尽量给出一个可以直接用于人工判断的静态事实清单：

- 哪些函数会直接崩
- 哪些依赖还没闭包到位
- 哪些位置只能判定为“不可信”
- 哪些寄存器间接调用需要你继续回溯
- 在整条调用链上，哪些地方出现了静态数据或绝对地址引用

当前脚本面向 `x86_64`，依赖 `pyelftools` 和 `capstone`。

## 2. 检查规则

### Rule 1: 特权环境嗅探

命中特权指令或 `CR0/CR2/CR3/CR4/CR8` 控制寄存器访问时，判定为 `HARD FAIL`。

### Rule 2: 绝对地址绞杀

命中 `R_X86_64_64`、`R_X86_64_32`、`R_X86_64_32S` 这类绝对地址相关重定位时，判定为 `HARD FAIL`。

### Rule 3: 空间几何测绘

对 `.rodata`、`.data`、`.bss` 等非可执行节区的引用进行记录。无论是相对寻址、绝对地址装载，还是其他可识别重定位，只要在调用链上出现，都会作为事实列出来：

- `READONLY`：当前节区不带写权限
- `WRITABLE`：当前节区带写权限

### Rule 4: 闭包边界审计

对 `call/jmp` 做闭包传播：

- 命中 Shim 白名单：视为用户态边界，停止向下追踪
- 命中其他 `.ko` 或 `vmlinux`：继续深入分析
- 彻底找不到：记为 `missing_deps`，结果变为 `INCOMPLETE`

### Rule 5: 间接控制流追踪

记录无重定位的寄存器 / 内存间接跳转，以及 Retpoline 包装跳转，提醒用户态在执行前补齐函数指针或跳转表。

对于 `__x86_indirect_thunk_rax` 这类 Retpoline 跳转，工具会直接从 thunk 符号名里提取目标寄存器，并在当前函数内回溯最近一次给该寄存器赋值的指令。

## 3. 分析结果语义

工具仍然给三种状态码，便于脚本化调用，但报告本身更偏向事实输出：

- `PASS` / exit code `0`
  闭包完整，未发现特权指令、控制寄存器访问或绝对地址重定位。
- `FAIL` / exit code `1`
  命中了明确的致命问题，继续复用大概率会崩。
- `INCOMPLETE` / exit code `2`
  没有命中致命问题，但存在闭包断裂、切片越界、空函数、反汇编失败等盲区，结果不能直接当作放行依据。

## 4. 日志与调试

当前脚本会输出更细的 trace 日志，便于调试闭包传播过程，例如：

- 当前深入分析的函数
- 新加入队列的依赖函数及来源
- 命中的 Shim
- 命中的插桩 / Retpoline
- 记录到的数据依赖
- 间接调用发生在哪个函数、哪条汇编、哪个运行地址
- 间接调用依赖哪些寄存器，以及这些寄存器在当前函数里最近一次写入来自哪里
- 如果是 Retpoline 调用，会根据 thunk 名称直接推断目标寄存器，例如 `__x86_indirect_thunk_rax -> rax`
- 缺失符号、节区越界、反汇编失败等告警

如果系统允许读取 `/proc/kallsyms`，报告里会额外显示内核运行时虚拟地址；否则退回静态地址展示。常规静态分析不要求 `sudo`，只有在某些启用了严格 `kptr_restrict` 的系统上，为了读取更完整的 `kallsyms` 才可能需要更高权限。

## 5. 依赖安装

```bash
pip3 install capstone pyelftools
```

此外需要准备未被 strip 的目标 `.ko` 与对应的 `vmlinux`。

## 6. 配置文件

### 6.1 Shim 配置

Shim 文件用于声明“用户态会自行兜底”的外部依赖 API。格式为按空白符分隔的 token，支持 `#` 注释。`-t` 指定的顶层目标始终分析真实实现；Shim 只截断分析过程中遇到的依赖调用。

示例：

```text
# memory
kmalloc kfree kcalloc

# libc-like
memcpy memset memmove

# logging
printk
```

### 6.2 额外插桩前缀配置

可选的 `inst-file` 用于补充本项目未内置的工具链插桩前缀，同样按空白符分隔。

示例：

```text
# local toolchain markers
__custom_sanitize_
__my_profile_
```

## 7. 使用方式

### 命令格式

```bash
python3 functions_checker.py \
  -m <模块1.ko> [模块2.ko ...] \
  -t <入口函数名> \
  -v <vmlinux路径> \
  [-s <shim.txt>] \
  [-i <inst_prefix.txt>]
```

### 示例

```bash
python3 functions_checker.py \
  -m my_zlib.ko helper.ko \
  -t my_zlib_deflate \
  -v /usr/lib/debug/boot/vmlinux-$(uname -r) \
  -s shim.txt
```

如果只想分析 `vmlinux` 内的目标函数，也可以不传 `-m`。

## 8. 报告解读

### `[❌ HARD FAILS - 物理层崩溃预警]`

列出明确不能放行的问题，例如：

- 特权指令
- 控制寄存器访问
- 绝对地址寻址

### `[⚠️ 分析不完整 / 结果不可信]`

列出当前工具无法可靠完成闭包的原因，例如：

- 丢失符号
- 函数 `size == 0`
- 节区切片越界
- 反汇编结果为空

这类结果不应被解释为“安全”，只能说明“目前还无法确认”。

### `[📦 静态数据引用事实]`

列出当前代码实际引用到了哪些非可执行节区，重点回答：

- 哪个函数里出现了这条指令
- 函数内偏移是多少
- 运行时虚拟地址是多少
- 汇编本身是什么
- 目标节区和符号/槽位是什么，例如 `.data+0x0`、`.data+0x20`
  对 RIP 相对重定位，报告会尽量展示修正后的实际槽位偏移，而不是原始 addend

- `READONLY`：节区本身不带写权限
- `WRITABLE`：节区本身带写权限

### `[🔌 Shim 边界成功拦截]`

列出被 Shim 文件成功兜底的符号调用点。

### `[🔀 间接控制流 / 盲跳]`

列出函数指针调用、寄存器盲跳、Retpoline 包装间接跳转等位置，并补充：

- 发生在哪个函数
- 函数内偏移和运行时虚拟地址
- 参与寻址的寄存器
- 当前函数内最近一次相关寄存器写入来自哪条汇编
- 若写入指令有关联重定位，会显示它依赖的节区、符号或 `section+offset`

## 9. TODO

### 9.1 间接跳转目标具体化

当前工具对间接跳转的处理，仍然停留在“识别盲跳并解释目标寄存器来源”的阶段。这个粒度对于纯 PIC 检查已经有价值，但对于“内核代码映射到用户态执行”的场景还不够。

典型问题是：

- 入口函数本身只通过函数指针发起调用，例如 `strm->zalloc(...)`
- 该函数指针在更早的初始化路径中被设置为模块内函数，例如 `zcalloc`
- `zcalloc` 又继续依赖 `kvmalloc_node`、`kvfree` 这类必须由 `libshim.so` 平替的内核 API

如果 checker 在间接跳转处就停止递归，只把它记成 “Retpoline 包装的盲跳”，那么这类真实依赖会被隐藏，导致：

- `functions_checker.py -t deflateInit2_` 可能通过
- 但 `functions_checker.py -t zcalloc` 才暴露出 `kvmalloc_node`

后续建议补三层增强：

1. `.data` / `.rodata` 槽位解析

- 如果间接跳转目标寄存器来自 `mov reg, [rip + ...]`
- 并且该读取落在 `.data` / `.rodata` / `.data.rel*` 等静态槽位
- 则继续结合 `.rela.data` / `.rela.rodata` 把槽位具体化为真实符号

2. store-load 传播

- 如果目标寄存器来自结构体字段或对象槽位，例如 `[r12 + 0x40]`
- 则尝试在当前函数或初始化路径中继续追踪：
  - 这个槽位之前是由谁写入
  - 写入值是否来自固定符号或伪 GOT 槽位
- 对本项目尤其重要，因为很多伪造 GOT 都会先把函数地址写进 `.data` 上下文或对象字段，再经由函数指针调用

3. 未被 Shim 的 kernel-only API 判定

- 当间接跳转最终被具体化到某个内核函数后，不应只继续普通闭包分析
- 还应额外判断：
  - 该目标是否为仅内核态可执行的函数
  - 该目标是否已被 shim 白名单声明为用户态替身
- 如果命中 kernel-only API 且未被 shim，应在报告中明确标记为高风险，必要时升级为 `FAIL` 或新增专门告警类目

这部分增强的目标不是做完整的 points-to analysis，而是优先覆盖最常见、最有价值的模式：

- 伪 GOT 槽位
- 对象字段中的固定函数指针
- 初始化函数里写死的默认 allocator / free callback

## 9. 当前限制

- 目前实现绑定 `x86_64`
- 对重定位类型的处理仍然偏保守，重点覆盖最常见场景
- 结果依赖目标 ELF 中保留完整符号表
- 这是静态分析工具，不会替代真实执行验证
