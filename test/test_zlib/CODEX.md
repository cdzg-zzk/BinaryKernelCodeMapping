# test_zlib 项目说明

## 目录职责

### `zlib_lkm/`
- 这个目录只负责一件事：把从 `libz.so` 提取出来的 `compress-only` 代码构建成 PIC/LKM。
- 这里是后续“同源实现进入内核”的主工作区。
- 目标不是做基准测试，也不是长期堆放实验结果，而是稳定地产出可注入内核、可被导出的实现。

### `so/`
- 这个目录负责与机制直接相关的动态库和中间产物。
- 这里用于放置需要参与 `page cache replace` 的 stub DSO、导出后的共享库以及相关辅助文件。
- 它承担的是“机制侧产物管理”职责，不承担论文实验的主分析职责。

### `pigz_macro/`
- 这个目录是后续论文实验主目录。
- 这里用于保存：
  - `pigz` 和 `zlib` 相关源码基线
  - 测试脚本
  - 输入语料
  - `perf` 报告
  - benchmark 结果
  - 实验过程文档
- 这里的重点是“做实验、留证据、形成论文可复现实验链路”。

## 项目目标

当前项目的核心目标已经调整为：

1. 从用户态 `libz.so` 中提取 `pigz` 实际需要的 `compress-only` 代码路径。
2. 将这份同源代码做成 PIC/LKM，并注入内核。
3. 通过 `page cache replace` 将其导出成机制侧可用的共享库。
4. 在用户态提供 `libmz.so` 这一层 ABI 兼容封装，使 `pigz` 可以不改源码直接复用。
5. 用同一份 `zlib` 源实现做基线和机制版本对比，从而尽量隔离“机制开销”和“实现差异”。

这次路线调整的核心原因是：
- 之前使用的 kernel-derived raw deflate 实现与上游 `libz.so` 并不是同一份实现。
- 即使功能一致，最终测到的性能差距也会混入“实现差异”。
- 对论文来说，这种对比不够干净，难以说明性能差距究竟来自机制本身还是压缩器实现本身。

因此，后续实验应尽量满足：
- 压缩算法实现同源
- 暴露 ABI 一致
- 差异尽量只保留在“是否经过内核映射/导出机制”这一层

## 实验背景

`pigz` 是当前实验宿主程序，因为它具备以下特点：

- 真实使用 `zlib` 的压缩路径，而不是自定义压缩器
- 支持单线程和多线程，便于观察不同工作负载
- 会使用 `deflate` 相关接口而不是只调用单一 `compress()`
- 适合做端到端吞吐、压缩率和热点分析

我们当前关注的是 `compression-only` 场景，而不是完整替代 `libz.so` 的全部 ABI。
也就是说，目标不是提供完整 `inflate`/`gz*` 兼容库，而是保证 `pigz` 的压缩路径能够正确运行并可用于性能评估。

## 宏观测试思路

宏观测试关注端到端表现，即“真实程序 + 真实输入”的性能和正确性。

推荐把宏观测试固定为以下几类：

### 1. 正确性回归
- 用 `pigz` 配合 `LD_PRELOAD=libmz.so` 运行压缩任务。
- 对输出使用：
  - `gzip -t`
  - `gunzip -c ... | cmp -s input -`
- 至少覆盖：
  - 单线程 `-p 1`
  - 多线程 `-p 4`
  - `-H`
  - `-i`

### 2. 吞吐/耗时测试
- 比较系统 `libz.so` 与机制导出的 `libmz.so`。
- 固定输入、线程数、压缩级别、重复次数。
- 记录：
  - real/user/sys 时间
  - 输出大小
  - 是否通过解压校验

### 3. 压缩率对比
- 同一输入上对比 `.gz` 输出大小。
- 如果源实现同源，那么压缩率差异应明显缩小。
- 若仍存在差异，应优先怀疑 wrapper 语义、导出机制副作用或状态同步问题。

## 微观测试思路

微观测试关注压缩器内部路径和机制引入的底层代价。

推荐固定为以下几个方向：

### 1. `perf stat`
- 看整体指令数、cycles、IPC、branch miss、cache miss。
- 目的：
  - 判断是“做了更多工作”，还是“同样工作做得更慢”。

### 2. `perf record/report`
- 识别热点函数和主要时间消耗位置。
- 需要尽量让符号可见，避免只有匿名偏移。
- 重点观察：
  - `deflate` 主循环
  - `fill_window`
  - `longest_match`
  - `trees` / Huffman 构造
  - CRC 路径
  - 机制相关导出/跳转路径

### 3. block / bitstream 分析
- 在必要时比较输出 bitstream 的内部结构。
- 目的不是只看“结果文件大小”，而是进一步判断：
  - block 切分是否一致
  - flush 语义是否一致
  - dictionary 作用后的编码行为是否一致

### 4. 单流与多流分离分析
- `pigz -p 1` 和 `pigz -p 4` 需要分别分析。
- 单流更适合观察压缩器本体。
- 多流更容易暴露 dictionary、边界拼接、stream 管理问题。

## 推荐实验路径

后续建议按下面顺序推进，避免同时改动太多层导致问题难以归因。

### 阶段 1：确定同源实现基线
1. 固定一份上游 `zlib` 版本作为唯一源码基线。
2. 明确 `pigz` 压缩路径实际需要的接口子集。
3. 从这份 `zlib` 中提取 `compress-only` 所需源码，而不是继续使用另一套 kernel-derived deflate。

### 阶段 2：在 `zlib_lkm/` 中构建 LKM
1. 将提取后的代码整理为适合 PIC/LKM 的构建形式。
2. 尽量保持源码逻辑与原始 `zlib` 一致，减少“适配性重写”。
3. 把为内核环境必须做的改动单独记录，便于论文说明。

### 阶段 3：在 `so/` 中生成机制侧 DSO
1. 准备参与 `page cache replace` 的 stub DSO。
2. 将内核中导出的实现映射为用户态可链接/可 preload 的共享库。
3. 固定导出符号、依赖关系和运行时路径。

### 阶段 4：在 `pigz_macro/` 中建立论文实验链路
1. 固定输入语料。
2. 固定运行命令。
3. 固定结果输出路径。
4. 固定 perf 采样方式。
5. 将所有实验命令和结果保存为文档，保证可复现。

## 论文中的对比原则

后续论文对比应尽量满足以下原则：

1. 基线版本和机制版本尽量来自同一份 `zlib` 源码。
2. `pigz` 本体不修改，或仅做最小必要改动。
3. 比较对象唯一主要差异应是“是否经过机制导出”。
4. 正确性优先于性能，任何性能结论都必须建立在正确输出之上。
5. 若为了兼容而在 wrapper 中做特殊处理，必须单独说明，避免把 wrapper 逻辑误写成机制收益。

## 建议保留的数据和文档

下面这些内容建议统一放在 `pigz_macro/` 中长期保留：

- benchmark 原始 CSV
- `perf stat` 文本结果
- `perf report` 文本导出结果
- 输入语料说明
- 运行命令记录
- 与每轮实验对应的 README/REPORT 文档

这样做的目的，是让后续任何窗口都能快速回答三个问题：

1. 当前基线是什么实现？
2. 当前机制版本是什么实现？
3. 当前性能差异到底来自哪一层？

## 后续窗口接手时的默认认知

后续窗口应默认按下面的理解继续工作：

- `zlib_lkm/` 负责“同源 zlib 代码 -> PIC/LKM”
- `so/` 负责“page cache replace 相关 DSO/stub/导出产物”
- `pigz_macro/` 负责“论文实验、宏观/微观测试、perf 与报告沉淀”

如果后续需要继续扩展目录，也应尽量保持这三层职责分离，避免源码构建、机制产物和实验数据混在一起。
