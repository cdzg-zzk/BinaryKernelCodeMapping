# VKSO native x86-64 final functional audit

## Result

The Linux 5.15.198 native x86-64 time/getcpu scope is functionally complete
under QEMU.  Raw vDSO and VKSO use configs that differ only in their
implementation selects, and their normalized semantic/path matrices are
identical:

- 76 single-vCPU semantic, fast/fallback and time-namespace rows.
- 7 four-vCPU getcpu/multi-thread rows.
- Two VKSO map/execute/restore cycles complete without a stale mapping.
- `CONFIG_VKSO_TIME=y` and `CONFIG_VKSO_TIME=n` both build.
- The production config has `CONFIG_VKSO_TIME_TEST=n`, no
  `__vkso_test_*` symbol and no test-probe object.

PVClock and Hyper-V are production-compiled, direct cold paths.  They have
static symbol and disassembly evidence but no runtime claim because the
available QEMU configuration does not expose those clock modes.  QEMU cycle
counts are not treated as performance evidence.

The authoritative per-case status is in
`linux-5.15.198-vkso/VKSO_FUNCTION_MATRIX.md`.

## Structure

The implementation is split into four production roles:

1. The kernel adapter publishes the minimal global shared snapshot and
   per-MM namespace offsets, injects kernel-valid counter context and selects
   kernel fallback.
2. The shared core owns seq validation, clock selection, cycle conversion and
   result calculation.  Kernel and user callers execute this implementation.
3. TSC remains inline in the hot core.  PVClock and Hyper-V are separate
   providers reached by direct calls only when their counter mode is selected.
4. The hand-written user wrapper obtains MM data from `AT_VKSO_MM_DATA`,
   hides the internal cycle context, adapts public types and performs a raw
   syscall fallback.

`getcpu`, counter providers and `CONFIG_VKSO_TIME_TEST` probe code are in
separate translation units.  The test probe is not linked into a production
build.  Generating the final `__vkso_*` wrapper symbols in `make_dll` is
intentionally deferred; the current `vkso_user_*` wrappers already validate
the required signatures, semantics and fallback policy.

## Fast and fallback paths

The seccomp-based path oracle proves that both raw vDSO and VKSO use the fast
path for:

- `clock_gettime`: realtime, monotonic, monotonic-raw, boottime, TAI and both
  coarse clocks.
- `clock_getres`: the same seven clocks, including valid `NULL` output.
- All four `gettimeofday` NULL/output combinations.
- `time(NULL)`, `time(&value)` and all `getcpu` output/cache combinations.

Both implementations fall back for process/thread CPU clocks, alarm clocks
and invalid IDs.  NULL and error behavior is compared in isolated child
processes where required, so matching faults as well as return values are
covered.

Time-namespace coverage includes exec, setns/commit, frozen offsets, auxv/MM
mapping continuity, five namespace-relevant fast clocks and TAI isolation.

## Cache-line analysis

The production build enforces the layout with `static_assert`:

- Bytes 0..63 contain seq/version, TSC conversion metadata and the realtime
  and monotonic bases.  Those two dominant high-resolution reads therefore
  use one hot 64-byte line for the coherent snapshot.
- The raw snapshot occupies bytes 128..167 and fits in one aligned line.
  Before the reorder, its data began at byte 96 and its base crossed the
  128-byte boundary.  A monotonic-raw reader now needs the seq line plus one
  raw-data line instead of a seq line plus a straddled raw region.
- The writer still publishes the complete snapshot under one seq generation.
  The reorder does not reduce writer invalidations or alter retry semantics;
  it only improves the raw reader's cache-line working set.

Disassembly confirms inline `rdtsc`, zero indirect calls in the shared time
core and direct relocations to both cold providers.  Actual latency and retry
rate remain bare-metal measurements, not QEMU conclusions.

## Code and text size

The values below are generated by `final_audit.sh`.  SLOC means nonblank,
non-comment source lines in the explicitly listed production/test files.
Compatibility is a changed-line delta against the clean no-vDSO kernel,
because counting whole pre-existing kernel files would be misleading.

| Category | Result |
|---|---:|
| Shared core | 481 SLOC |
| Counter providers | 39 SLOC |
| Kernel adapter/MM publication | 387 SLOC |
| Hand-written user wrapper and ABI header | 140 SLOC |
| Test/probe code | 3152 SLOC |
| Existing-kernel compatibility integration | +192 / -40 lines |
| VKSO shared-core `.vkso.text` | 1029 bytes |
| VKSO counter-provider `.vkso.text` | 172 bytes |
| VKSO getcpu `.vkso.text` | 38 bytes |
| Total mapped VKSO production text | 1239 bytes |
| Raw vDSO complete `.text` | 1446 bytes |
| Raw vDSO five exported API bodies | 1223 bytes |
| User wrapper five steady API bodies | 179 bytes |
| User wrapper complete `.text` including initialization/alignment | 313 bytes |
| VKSO mapped text + steady wrapper API bodies | 1418 bytes |

The mapped production text is 207 bytes (14.3%) smaller than the whole raw
vDSO text.  Including the five steady wrapper bodies gives 1418 bytes, 28
bytes (1.9%) below raw vDSO `.text`.  If the one-time wrapper initializer and
section padding are also charged, the total is 1552 bytes, 106 bytes (7.3%)
larger.  These are code-size observations, not speed claims.

## Reproduction

From the repository root, one command can rebuild and rerun the raw/VKSO QEMU
matrices, build the disabled configuration and regenerate all static
evidence:

```sh
RUN_QEMU=1 BUILD_OFF=1 \
  test/test_gettime/vkso-tests/m27/final_audit.sh
```

With existing build and QEMU artifacts, run:

```sh
test/test_gettime/vkso-tests/m27/final_audit.sh
```

The summary is written to `/tmp/vkso-m27-run/final_audit.txt`; normalized
matrices, diffs, disassembly and wrapper objects are in
`/tmp/vkso-m27-run/final-audit/`.

The only remaining scoped work is automatic wrapper integration in
`make_dll` and controlled bare-metal core/wrapper performance measurement.
