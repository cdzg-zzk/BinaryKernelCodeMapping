# Manual ELF Builder Technical Report

## Background

We started with `build_so.py`, a handcrafted ELF64 shared-object generator used to turn kernel
symbols into a synthetic, linkable dynamic library. The initial version relied on a `symbols.txt`
file that already contained absolute addresses, sizes, and symbol kinds. It emitted a single
`.text` section, a single `.rodata` section, and packed every live page from the minimum text
address up to the last read-only page, including huge gaps that rarely contained meaningful data.

Two practical requirements forced us to evolve the implementation:

1. **Accurate kernel metadata** – The authoritative address and dependency data lives in the
   `out.krg` graph generated by `krg.cpp`. The old script duplicated that information manually,
   quickly becoming stale.
2. **Sparse section emission** – Kernel text ranges often contain megabytes of unused space.
   Serializing those holes into the `.so` wastes disk space and forces the dynamic loader to map
   empty pages. We wanted the generated binary to include only the pages that contain useful data,
   while preserving the original virtual-address spacing for relocations and debugging.

## Key Challenges

### 1. Querying `out.krg` from Python

`krg.cpp` stores the SAFE graph in a compact custom format (`FileHeader`, adjacency lists, and a
null‑terminated name blob). We had to rebuild the “query” half of `krg` inside Python without
pulling in heavy external dependencies.

**Solution:** Reimplemented a minimal `KrgGraph` reader in Python (`build_so.py:167`). It parses the
header, node table, CSR adjacency, and name blob, then exposes a `closure()` method that mirrors the
C++ query behavior. The resolver reads bare symbol names from `symbols.txt`, looks up each symbol in
`out.krg`, performs a DFS/BFS to gather the full dependency closure, and differentiates between
explicitly requested symbols (exported) and implicit dependencies (internal only). Missing symbols no
longer abort the build; the resolver prints a warning and skips them unless *all* requested symbols
fail.

### 2. Keeping exported and internal symbols separate

Only the names listed in `symbols.txt` should surface as global entries in `.dynsym`/`.dynstr`.
Dependencies are needed to fill the `.text`/`.rodata` image, but they must stay hidden.

**Solution:** Each `Symbol` now tracks:

- whether it is exported,
- its relative order in the request list (to keep `.gnu.hash` deterministic),
- the section it belongs to (set later by the region splitter), and
- its computed virtual address.

Only exported symbols participate in `.dynsym`, `.dynstr`, and `.gnu.hash`, while the full symbol list
feeds the `.symtab` and section writer. This guarantees that dependencies benefit from a consistent
address space without leaking into the shared library’s ABI surface.

### 3. Removing text/rodata holes without breaking addresses

Kernel code and read-only data frequently occupy sparse page ranges. If we emit a single `.text`
section spanning from the lowest text address to the highest, every in-between page must be written
to disk and later mapped, even when it contains no symbols. We needed to serialize *only* populated
pages yet keep the original relative virtual addresses intact.

**Solution:** Introduced a `Region` abstraction and a page-aware splitter:

1. For each class (code/data), `_build_regions()` collects per-symbol page spans, merges overlapping
   runs, and produces contiguous regions. When multiple runs exist, it assigns names `.text1`,
   `.text2`, … or `.rodata1`, etc.
2. `_assign_symbol_sections()` records the owning region name for every symbol.
3. `_build_content_sections()` emits an ELF section per region, sized exactly to the region’s page
   span and filled with stubs (`0xC3`) or zeros.
4. `_assign_offsets_and_addresses()` writes those sections sequentially in the file but preserves
   their virtual addresses by anchoring each region to `TEXT_BASE_VADDR + (region.start -
   text_min_page)`. This keeps the gaps in virtual space without serializing them.
5. `_build_program_headers()` emits a PT_LOAD entry for every region so the loader knows precisely
   which pages to map (no more `PROT_NONE` gaps for empty spans).

The result is a sparse ELF where `.text*`/`.rodata*` sections mirror only the real pages, yet symbol
addresses remain exactly where callers expect them.

## Outcome

- **Input simplicity** – `symbols.txt` now lists only symbol names. Everything else (address, size,
  kind, dependencies) is derived automatically from the latest `out.krg`.
- **Robust exports** – Requested symbols become the only globals in the dynamic symbol table; all
  transitive dependencies are internal but still mapped into `.text*`/`.rodata*`.
- **Space savings** – Each non-contiguous region becomes its own section and program header, so the
  generated `.so` contains zero padding only where data exists. Example: instead of one `.text`
  covering `0x1000–0x74000`, we now emit `.text1`, `.text2`, `.text3`, etc., eliminating tens of
  megabytes of blank bytes.
- **Loader-friendly** – Multiple PT_LOAD entries are valid ELF and let the dynamic loader map only
  populated pages. Tools that iterate PT_LOAD now see explicit regions and don’t have to infer gaps.
- **Extensibility** – The code is modular: `KrgGraph` handles graph I/O, `resolve_requested_symbols`
  handles exports, the region splitter handles sparse layout, and the existing `ManualElfBuilder`
  handles ELF emission. Future tweaks (e.g., RW segments per page, RELRO, PIE adjustments) can be
  layered on top without touching unrelated pieces.

## Advantages for Research/Publication

1. **Accurate kernel mirroring** – We preserve real kernel virtual-address relationships, which is
   crucial when mapping in-kernel pointers or patch sites.
2. **Minimal binary size** – Sparse sections deliver huge file-size reductions, making experiments
   faster and easier to distribute.
3. **Deterministic exports** – The exported symbol set and ordering are fully defined by the input
   name list, simplifying reproducibility in papers and tooling.
4. **Pure Python workflow** – The entire build pipeline works without recompiling `krg.cpp` or
   relying on capstone/ELFIO, which keeps the environment lightweight and reproducible.

This architecture allowed us to iteratively solve each challenge: first by teaching the Python
script to query `out.krg`, then by separating exports from dependencies, and finally by developing a
region-based layout engine that faithfully reproduces the kernel’s sparse address space without
wasting space or breaking compatibility.
